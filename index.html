실수 #2: LLM의 실수를 LLM에게 직접 수정 요청

Don't: LLM의 실수를 직접 수정 요청

커서가 생성한 코드나 내용에 오류가 있을 때, 단순히 "이거 틀렸어, 고쳐줘" 라고 다시 요청합니다.

오류 수정 요청 (잘못된 방식)

특정 함수 만들어줘

function myFunction() {
  // 오류 있는 코드 부분
}

이거 틀렸어, 고쳐줘!

function myFunctionFixed() {
  // 또 다른 오류 발생
}

Do: 원인 프롬프트 수정 후 재요청

커서가 실수한 결과물을 만든 원인 프롬프트를 찾아 명확히 수정 후 다시 요청합니다.

프롬프트 수정 (올바른 방식)

특정 함수 만들어줘 (초기)

function myFunction() {
  // 오류 발견!
}
(코드 롤백 & 프롬프트 수정됨)

정확한 매개변수 포함한 함수 (수정)

function myFunction(param1, param2) {
  // 정상 작동하는 코드
}

Tip: 대부분의 커서 인터페이스에서 이전 프롬프트 수정 시 코드가 해당 시점으로 롤백되어 편리합니다. 근본 원인 수정이 효과적입니다.

Why? LLM은 다음 토큰 생성기이며, 커서는 잘못된 맥락에서 실수를 반복하거나 수정 의도를 파악하기 어렵습니다.

Don't: 수정 대상 명시 없이 지시

코드베이스의 어느 부분을 수정해야 하는지 알려주지 않고, "로딩 스피너 추가해줘" 또는 "API 에러 처리 개선해줘"와 같이 모호하게 요청합니다.

?

Do: 수정 대상 코드 명시 (파일 첨부)

수정할 코드 부분이나 관련 파일을 명확히 전달하여 컨텍스트를 제공합니다.

파일 첨부로 컨텍스트 제공

Why? 커서의 출력은 입력 정보에 의존합니다. 수정 위치나 컨텍스트가 없으면 커서가 추측하기 어렵고, 엉뚱한 결과를 낼 수 있습니다.

실수 #4: 린터 오류가 있는 상태에서 요청하기

Don't: 린터 오류 무시하고 요청

코드에 린터(Linter) 오류가 표시된 상태에서 커서에게 코드 생성이나 수정을 요청합니다.

린터 오류 + 요청
function greet() {
console.log("Hello") // Missing semicolon
}

이 함수에 파라미터 추가해줘.

세미콜론이 빠진 것 같아요. 먼저 린트 오류를 수정하는 것이 어떨까요? 그 후에 파라미터를 추가해 드릴게요.

Do: 린터 오류 해결 후 요청

요청 전 린터 오류를 먼저 해결하세요. 커서가 의도와 다르게 수정하면 포맷터 설정을 확인하세요.

린터 오류 해결 + 요청
function greet() {
console.log("Hello");
}
(Format on Save: ON)

이 함수에 파라미터 추가해줘.

function greet(name) {
  console.log(`Hello, ${name}`);
}

Tip: 'Default Formatter' 및 'Format on Save' 설정으로 코드 스타일을 일관되게 유지하세요. 사소한 오류로 커서를 혼란시키는 것보다 낫습니다.

Why? 린터 오류는 커서 컨텍스트에 포함됩니다. 사소한 오류에도 커서는 민감하게 반응해 핵심보다 오류에 집중하거나 잘못 추론할 수 있습니다.

실수 #5: 디자인 방향 없이 큰 작업 요청

Don't: 디자인 방향 없이 큰 작업 요청

명확한 디자인 계획이나 목적 없이 커서에게 "멋진 SNS 만들어줘"와 같이 광범위한 작업을 요청합니다.

광범위한 요청

멋진 SNS 서비스 만들어줘!

(혼란) 어떤 기능을 원하시나요? 사용자 흐름, 데이터 모델은요?

???

Do: 디자인 선행 후 단계별 작업 요청

먼저 커서에게 디자인(요구사항, 규칙, 마크다운 문서 등)을 요청하고, 이를 검토/수정한 후, 구체적인 태스크 단위로 작업을 지시합니다.

단계별 접근

1. User: SNS 디자인 문서(MD) 작성해줘

(기능: 포스팅, 댓글)

커서: 디자인 문서입니다. (검토/수정)

2. User: (검토/수정 후)

@SNS_디자인_문서 기반 포스팅 API 구현

커서: 포스팅 API 구현 중...

감이 안 잡히면: Deep Research 활용

Tip: 사용할 API, 시스템 흐름 등에 대한 지식과 디자인 결정이 선행되어야 합니다. 큰 작업에 대한 감이 없다면, Gemini/OpenAI Deep Research 등으로 베스트 프랙티스를 찾아보는 것도 도움이 됩니다.

Why? 큰 작업은 목적이 명확하지 않으면 사용자 의도나 프로젝트 구조와 벗어날 가능성이 커져 결국 만족스럽지 못한 결과로 이어집니다.

실수 #6: 기술 부채 방치하기

Don't: 기술 부채 방치

명확하지 않은 파일/변수/함수 이름, 부족한 모듈화 등 기술 부채가 쌓인 코드에 대해 커서에게 작업을 요청합니다.

기술 부채 + 요청

이 코드에 새 기능 추가해줘.

(혼란) 코드가 복잡해서 어디를 수정해야 할지 모르겠어요...

Do: 커서에게 코드 리팩토링 요청

커서에게 리팩토링을 요청해서 코드의 퀄리티를 유지합니다.

리팩토링 요청

@복잡한_코드.js 이 파일 SRP 준수해서 리팩토링하고, 모듈화 해줘.

리팩토링 완료!

Tip: 파일 하나만 지정하고 "이 파일 수정해줘"라고 했을 때 커서가 어디를 수정할지 명확히 알 수 있도록 코드를 관리하는 것이 이상적입니다.

Why? 커서는 파일 이름, 변수/함수 이름, 모듈 구조 등을 통해 코드를 이해합니다. 기술 부채가 쌓인 코드는 커서를 혼란스럽게 만들어 작업 효율을 떨어뜨립니다.

실수 #7: 만족스러운 결과물에 안주하고 계속 작업

Don't: 만족 후 바로 다음 작업 진행

커서가 만족스러운 코드나 결과물을 생성했을 때, 이를 저장하거나 커밋하지 않고 곧바로 다음 작업을 이어갑니다.

만족 후 저장 없이 진행
커서: function perfectCode() {
// 완벽한 로직!
}
👏
User: 좋아! 그럼 이제 다른 기능 추가해줘.
커서: function messedUp() {
// 이전 코드 어디갔지? 새로운 오류!
}
이전 작업 손실!

Do: 만족스러운 결과물이 나오면 즉시 커밋

만족스러운 결과물이 나왔을 때는 즉시 작업을 커밋(저장)하여 좋은 상태를 보존합니다.

만족 시 즉시 커밋
커서: function perfectCode() {
// 완벽한 로직!
}
User: 좋아! 바로 커밋하자.
Source Control
feat: Implement perfectCode function
Commit ✨
안전하게 저장!

Tip: 커밋 메시지는 Source Control 패널의 ✨ sparkle 버튼을 클릭하면 자동 생성됩니다. 좋은 상태를 놓치지 말고 바로 저장하세요.

Why? 좋은 결과물을 저장해두지 않으면 이전의 완벽했던 상태를 잃어버릴 수 있습니다. 커서의 체크포인트 기능이 항상 완벽하게 작동하는 것은 아니며, 새로운 작업 중 예상치 못한 변경으로 원래 잘 작동하던 코드를 망가뜨릴 위험이 있습니다.

Pro Tip #2: 코드 설명은 커서에게

이해 안되면 바로 질문하세요!

커서가 작성한 코드가 이해되지 않으면, 주저 말고 설명을 요청하세요.

코드 설명 요청
const x = data.map(d => d*2).filter(d > 10);

이 코드 왜 이렇게 작성했어? 설명해줘.

이 코드는 먼저 data 배열의 각 요소를 2배로 변환한 후, 그 결과 중 10보다 큰 값만 필터링합니다. 가독성과 성능을 고려하여 메서드 체이닝을 사용했습니다.

(또는) 앗, filter 조건에 오류가 있었네요! d => d > 10 으로 수정하는 것이 맞습니다.

기대 효과: 명쾌한 설명을 듣거나 커서가 스스로 실수를 수정하기도 합니다. 대화를 통해 더 깊이 학습할 수 있습니다.
