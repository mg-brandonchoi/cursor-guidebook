Mistake #2: Asking the LLM to Fix Its Own Mistakes Directly

Don't: Ask the LLM to fix its own mistake

When the code or content generated by Cursor has an error, you simply ask, "This is wrong, fix it."

Error Correction Request (Wrong Way)

Create a specific function for me.

function myFunction() {
  // Code with an error
}

This is wrong, fix it!

function myFunctionFixed() {
  // Generates another error
}

Do: Correct the original prompt and request again

Find the prompt that caused the error, correct it with more clarity, and then request again.

Prompt Correction (Correct Way)

Create a specific function (Initial)

function myFunction() {
  // Error found!
}
(Code rolled back & prompt edited)

Function with correct parameters (Corrected)

function myFunction(param1, param2) {
  // Correctly working code
}

Tip: In most Cursor interfaces, editing a previous prompt conveniently rolls back the code to that point. Correcting the root cause is most effective.

Why? An LLM is a next-token predictor. Cursor may repeat its mistake in a flawed context or struggle to understand the intention behind the correction.

Mistake #3: Giving Instructions Without Specifying the Target

Don't: Give instructions without specifying a target

Vaguely asking "Add a loading spinner" or "Improve API error handling" without specifying which part of the codebase to modify.

Vague Instruction

Add a loading spinner.

To which component should I add the loading spinner? Please specify if it's during data loading, on a button click, etc.

?

Do: Specify the target code (attach files)

Clearly provide the relevant code or files to give context for the modification.

Providing Context with File Attachment
1const [loading, setLoading] = useState(false);
2const fetchData = async () => {
3setLoading(true);
4// API call...
5};
UserList.jsx

Drag files here

UserList.jsx

Add a loading spinner UI to this component.

Got it! I'll add a loading spinner to UserList.jsx.

Tip: Accurately provide context by dragging and dropping files or using @filename. Selecting a code block and pressing Cmd+L is also useful.

Why? Cursor's output depends on its input. Without a specific location or context, it's hard for Cursor to guess, which can lead to incorrect results.

Mistake #4: Making Requests with Linter Errors Present

Don't: Ignore linter errors and make a request

Requesting code generation or modification from Cursor while there are visible linter errors in the code.

Linter Error + Request
function greet() {
console.log("Hello") // Missing semicolon
}

Add a parameter to this function.

It looks like a semicolon is missing. How about fixing the lint error first? I can add the parameter for you after that.

Do: Fix linter errors before making a request

Fix linter errors first. If Cursor modifies code in an unintended way, check your formatter settings.

Linter Error Fixed + Request
function greet() {
console.log("Hello");
}
(Format on Save: ON)

Add a parameter to this function.

function greet(name) {
  console.log(`Hello, ${name}`);
}

Tip: Keep your code style consistent with 'Default Formatter' and 'Format on Save' settings. It's better than confusing Cursor with minor errors.

Why? Linter errors are included in Cursor's context. Cursor is sensitive to even minor errors, which can cause it to focus on them instead of the main task, or to make incorrect inferences.

Mistake #6: Neglecting Technical Debt

Don't: Neglect Technical Debt

You ask Cursor to work on code with accumulated technical debt, such as unclear file/variable/function names and poor modularization.

Technical Debt + Request

Add a new feature to this code.

(Confused) The code is too complex, I don't know where to modify it...

Do: Ask Cursor to Refactor Code

Request refactoring from Cursor to maintain code quality.

Refactor Request

@complex_code.js Please refactor this file by following SRP and modularize it.

Refactoring complete!

Tip: It's ideal to maintain your code in a way that makes it clear what to change when you say "modify this file".

Why? Cursor understands code through file names, variable/function names, and module structure. Code with technical debt confuses Cursor and reduces its productivity.

Mistake #7: Continuing to Work After Getting a Satisfactory Result

Don't: Proceed to the next task immediately after getting a satisfactory result

When Cursor generates satisfactory code or results, you continue with the next task without saving or committing.

Proceeding without saving after satisfaction
Cursor: function perfectCode() {
// Perfect logic!
}
👏
User: Great! Now add another feature.
Cursor: function messedUp() {
// Where did the previous code go? New error!
}
Previous work lost!

Do: Commit immediately when you get a satisfactory result

When you get a satisfactory result, commit the work immediately to preserve the good state.

Commit Immediately on Satisfaction
Cursor: function perfectCode() {
// Perfect logic!
}
User: Great! Let's commit right away.
Source Control
feat: Implement perfectCode function
Commit ✨
Saved safely!

Tip: A commit message can be auto-generated by clicking the ✨ sparkle button in the Source Control panel. Don't lose a good state; save it immediately.

Why? If you don't save a good result, you can lose the previously perfect state. Cursor's checkpoint feature doesn't always work perfectly, and there's a risk of breaking previously working code with unexpected changes during a new task.

Pro Tip #2: Ask Cursor for Code Explanations

If you don't understand, ask right away!

If you don't understand the code Cursor has written, don't hesitate to ask for an explanation.

Requesting Code Explanation
const x = data.map(d => d*2).filter(d > 10);

Why did you write the code this way? Explain it to me.

This code first transforms each element of the data array by doubling it, and then filters for values greater than 10. I used method chaining for readability and performance.

(Or) Oops, there was an error in the filter condition! It should be corrected to d => d > 10.

Expected Effect: You might get a clear explanation, or Cursor might even correct its own mistake. You can learn more deeply through conversation.

Pro Tip #3: Natural Language Code Search

"Where was that function again?" Just ask Cursor!

If you can't remember a function name or file location, ask Cursor in natural language instead of searching.

Natural Language Code Search

Find the controller code that handles user login.

Searching...

Found the `handleLoginRequest` function in the `AuthController.js` file.

function handleLoginRequest(req, res) {
  // ... login processing logic ...
}

Advantage: You can find code as if asking a colleague, even without knowing the keywords, which is especially useful in large, complex projects.

Pro Tip #4: New Library? Boost Accuracy with Context7 MCP!

Let Cursor read the official docs!

When Cursor doesn't know the latest library information, provide the official documentation using tools like Context7 MCP to improve accuracy.

Context7 MCP (Doc Referencing)
User: Modify this part with the latest SwiftUI API.
Cursor: (Uses outdated API) ...
User: (Provides SwiftUI official docs via Context7 MCP)
Cursor: (Uses latest API) ...
(Reduced hallucination, improved accuracy)

Effect: It guides Cursor to generate accurate code based on the latest information, reducing hallucinations and improving the quality of answers (e.g., responding to SwiftUI API changes).

Mindset for Handling AI Tools

Code Review is NON-NEGOTIABLE!

Even if Cursor generated the code, you must review and understand it yourself. Do not blindly trust auto-generated code.

Don't let your skills get rusty!

AI is a learning tool, not just a replacement for simple tasks. Strive to understand the core logic and learn through AI.

Quality in = Quality out.

You can only get high-quality results by providing clear prompts and sufficient context (the fuel) and giving the right direction (the steering).

Cursor is a Super-Smart Co-Pilot

Cursor is a very smart co-pilot, but it is not yet an autonomous agent. The final judgment and responsibility lie with the user.
